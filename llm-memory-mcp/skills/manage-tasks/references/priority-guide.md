# 优先级判断指南

## 优先级定义

| 级别 | 名称 | 图标 | 响应时间 |
|-----|------|------|---------|
| 4 | 紧急 | 🔴 | 立即/24h内 |
| 3 | 高 | 🟠 | 1-3天 |
| 2 | 中 | 🟡 | 本周 |
| 1 | 低 | 🟢 | 有空再做 |

## 判断标准

### 🔴 优先级 4（紧急）

**关键词**：Bug、阻塞、崩溃、安全、紧急、立刻

**场景**：
- 生产环境 Bug
- 系统崩溃/不可用
- 安全漏洞
- 阻塞其他工作
- 客户紧急需求
- 24 小时内必须完成

### 🟠 优先级 3（高）

**关键词**：重要、尽快、影响体验

**场景**：
- 重要功能开发
- 影响用户体验的问题
- 3 天内需要完成
- 多人依赖的任务
- 即将到期的需求

### 🟡 优先级 2（中）- 默认

**关键词**：正常、常规、本周

**场景**：
- 常规功能开发
- 文档更新
- 代码优化
- 本周内完成即可
- 不紧急的改进

### 🟢 优先级 1（低）

**关键词**：可选、有空、技术债、未来

**场景**：
- 技术债务清理
- 可选的改进
- 长期优化计划
- 实验性功能
- 学习/调研任务

## 快速判断流程

```
任务来了
    ↓
是否影响生产/安全？ → 是 → 🔴 紧急
    ↓ 否
是否阻塞其他工作？ → 是 → 🔴 紧急
    ↓ 否
是否影响用户体验？ → 是 → 🟠 高
    ↓ 否
是否有明确截止日期？
    ↓
  24h内 → 🔴 紧急
  3天内 → 🟠 高
  本周内 → 🟡 中
  无期限 → 🟢 低
```

## 示例

| 任务 | 优先级 | 原因 |
|------|--------|------|
| 登录页面崩溃 | 🔴 4 | 用户无法使用 |
| SQL 注入漏洞 | 🔴 4 | 安全问题 |
| 新增支付功能 | 🟠 3 | 重要功能 |
| 优化查询性能 | 🟡 2 | 常规优化 |
| 更新 README | 🟢 1 | 有空再做 |
